## 5.  Docker e contexto em que foi criado

Para começar, é importante entender a história dessa tecnologia, pois não dá para falar sobre o Docker sem mencionar a Docker Inc. Então, vamos explorar de onde essa história veio.

Tudo começou em 2013, quando uma empresa chamada DotCloud precisava criar uma ferramenta para rodar as aplicações dos seus clientes. A DotCloud era uma Platform as a Service (PaaS), basicamente um concorrente do Heroku, comprado pela Salesforce, onde os desenvolvedores não precisavam se preocupar com infraestrutura para subir suas aplicações. A solução era simples: o desenvolvedor fazia o upload da aplicação, e ela começava a rodar.

Para viabilizar isso, a DotCloud criou o Docker em 2013. Quando o Docker Engine, ferramenta para rodar containers, foi disponibilizado no mercado, houve uma grande repercussão. Empresas do mundo todo começaram a se interessar pela tecnologia, que rapidamente se tornou popular. Esse sucesso levou a DotCloud a pivotar suas operações, abandonando o modelo de PaaS e renomeando-se para Docker Inc.

É interessante notar que o Docker surgiu para resolver uma necessidade específica da DotCloud, mas acabou se tornando uma ferramenta open source de grande relevância. Hoje, com mais de 10 anos, o Docker é uma solução madura que já cumpriu o propósito inicial para o qual foi criado.

### Docker Community Edition vs Docker Desktop
Um ponto importante a ser destacado é a diferença entre o Docker Community Edition e o Docker Desktop. O Docker Community Edition, também conhecido como Docker Engine, é open source e permite rodar containers sem custos, enquanto o Docker Desktop é um produto da Docker Inc., voltado para o desenvolvimento. Ela não é uma ferramenta open source, sendo projetada para ajudar no processo de desenvolvimento, mas não para atividades de DevOps. É, inclusive, uma das formas pelas quais a Docker Inc. gera receita.

### Docker
O Docker, para funcionar, aproveita recursos do próprio Linux. Isso é interessante pois é como mencionei anteriormente você aproveita os recursos do sistema operacional para conseguir fazer executar containers. Além disso, a Docker Inc. desmembrou alguns de seus componentes, como o RunC e o ContainerD, e os doou para a Open Container Initiative (OCI). Esses componentes são runtimes de baixo nível usados no gerenciamento de containers. O Docker, por outro lado, é considerado um runtime de mais alto nível, projetado para ser fácil de usar, agregando diversas funcionalidades em uma única ferramenta.

Antes do Docker, soluções como o LXC já existiam, mas eram complexas e menos eficientes. A grande ideia da DotCloud foi criar uma solução de alto nível que qualquer pessoa pudesse usar com facilidade. O Docker consolidou-se como uma ferramenta completa, capaz de gerenciar containers, trabalhar com redes, volumes e criar builds de imagens reutilizáveis. Então, o Docker é uma ferramenta 360 graus que gerencia o container.

O Docker opera no formato client-server. Quando você usa o comando docker no terminal, está interagindo com a interface de linha de comando (command-line interface), que funciona como um cliente. Cada comando é enviado para um servidor, o docker-d, que é o daemon responsável por gerenciar todo o ecossistema de containers. Esse daemon centraliza o gerenciamento, mas também representa um ponto único de falha (SPOF - Single Point of Failure). Se o docker-d cair, todos os containers gerenciados por ele também caem. Outras soluções, como o Podman, evitam esse tipo de problema, já que cada container funciona como um processo isolado.

Outro aspecto relevante é que o Docker, por padrão, requer permissões de superusuário (root) para ser executado. Essa característica pode ser explorada por vulnerabilidades e representa um risco potencial. Contudo, o Docker também pode ser configurado para funcionar em modo rootless, onde utiliza apenas o usuário local, eliminando a dependência de privilégios elevados.
