# Publicando Portas

Para que um serviço rodando dentro de um container seja acessível pelo seu computador (host), precisamos mapear as portas do container para as portas da nossa máquina.

## Entendendo a Exposição de Portas

Ao rodar um container de Nginx (`docker run nginx`), ele expõe a porta **80** por padrão. No entanto, se você tentar acessar `http://localhost:80` no seu navegador, a conexão falhará.

**Por que isso acontece?**
A porta 80 está aberta **dentro** do container. O container funciona como uma máquina isolada com seu próprio IP e rede. Para o seu computador "enxergar" esse serviço, é necessário criar uma ponte (redirecionamento).

## Publicando Portas com a flag -p

O parâmetro `-p` (publish) faz esse mapeamento. A sintaxe é sempre:
`PORTA_DO_HOST : PORTA_DO_CONTAINER`

```bash
# Mapeia a porta 8080 do seu PC para a porta 80 do container
docker run -p 8080:80 nginx
```

Agora, ao acessar `localhost:8080` no seu browser, o Docker redireciona a chamada para a porta 80 interna do container Nginx.

### Portas Baixas vs. Portas Altas
- **Portas Baixas (0-1023):** Exigem privilégios de superusuário (root) no sistema operacional para serem abertas.
- **Portas Altas (1024+):** Podem ser usadas livremente por usuários comuns.
Por isso, em desenvolvimento, é comum usarmos portas como `8080`, `3000` ou `9000`.

## Verificando o Redirecionamento

Ao rodar `docker ps`, você verá a configuração na coluna **PORTS**:
`0.0.0.0:8080->80/tcp`

Isso confirma que qualquer chamada para o seu IP (ou localhost) na porta 8080 será enviada para a porta 80 do container.

## O Conceito de Efemeridade

Um ponto crucial sobre containers é que eles são **efêmeros**. 

> [!CAUTION]
> **Containers não são Máquinas Virtuais (VMs):** 
> Em uma VM, os arquivos persistem no disco físico. No Docker, as alterações feitas dentro do container ficam em uma camada temporária (muitas vezes em memória via TMPFS).

**O que isso significa na prática?**
Se você entrar no container (`docker exec`), criar arquivos ou alterar configurações, e depois remover esse container (`docker rm`), **todos os dados serão perdidos permanentemente**. Ao criar um novo container da mesma imagem, ele virá "limpo", sem suas modificações.

### Como persistir dados?
Para não perder seu trabalho ou banco de dados, precisamos de mecanismos de persistência que conectem o sistema de arquivos do container ao disco rígido do seu computador. Este é o assunto do próximo capítulo sobre **Volumes**.
